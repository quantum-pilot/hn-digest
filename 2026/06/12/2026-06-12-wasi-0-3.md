# WASI 0.3

- Score: 257 | [HN](https://news.ycombinator.com/item?id=48504063) | Link: https://bytecodealliance.org/articles/WASI-0.3

### TL;DR
WASI 0.3 is now ratified and rebases the WebAssembly System Interface on the WebAssembly Component Model’s new async primitives. Instead of each component running its own event loop, the host now manages a shared async runtime using `stream<T>`, `future<T>`, and `async func`, greatly simplifying earlier WASI 0.2 patterns. This enables idiomatic async bindings across languages (Rust, Go, JS, Python, C#, C) and lets `wasi:http` support efficient in-process service chaining, turning cross-microservice calls from milliseconds into nanoseconds.

### Comment pulse
- Concern: process felt opaque and slow → progress and dynamic loading status were hard to track, making some wary of WASI components’ viability — counterpoint: meetings, repos, and talks are public but poorly narrated.  
- Dispute over goals: some want dynamic hot-loading and arbitrary linking; others argue the real target is controlled plugin ecosystems with fixed shapes and sandboxed, language-agnostic plugins.  
- Critique: component model is overcomplicated versus a simple POSIX-style WASI; defenders say it’s optional, not core wasm, and necessary for safe multi-language interop via IDL-like WIT.

### LLM perspective
- View: 0.3 shifts WASI from “Unix-like shim” toward a typed, async, multi-language component fabric; that’s a strategic change, not just a version bump.  
- Impact: runtimes, FaaS platforms, and plugin-heavy systems gain a credible, standardized way to host sandboxed, async components across languages.  
- Watch next: quality of dev tooling, dynamic loading patterns, real-world benchmarks for `wasi:http` service chaining, and whether major clouds adopt WASI 0.3 as a target.
