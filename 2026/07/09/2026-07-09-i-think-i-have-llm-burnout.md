# I think I have LLM burnout

- Score: 381 | [HN](https://news.ycombinator.com/item?id=48839984) | Link: https://www.alecscollon.com/blog/llm-burnout/

- TL;DR
    - Commenters describe “LLM burnout” as machine-driven pressure: tools can generate infinite tickets, docs, and code, expanding surface area without expanding human attention. Coding agents and chat assistants increase context switching, low-quality output to review, and a homogenized, grating prose style. Some report 10–20x productivity and ambitious solo projects, yet feel mentally and even physically exhausted, likening this shift to industrial automation. Themes: mandated usage, shallow understanding, inflated expectations, and calls for norms and boundaries around when and how to use LLMs.

- Comment pulse
    - LLMs inflate workload and pressure → they create endless docs, plans, and code to “unblock,” plus metrics and leaderboards that penalize not using them.
    - Coding agents erode deep work → devs juggle many agent chats, verify shaky correctness, and review 10x more low-quality PRs, straining attention and team culture.
    - LLM outputs feel alien and numbing → repetitive dialect, misused jargon, and sameness of voice make some readers queasy, while collaborators rely on them uncritically.

- LLM perspective
    - View: Burnout stems from unconstrained automation plus human perfectionism and fear-of-missing-out, not from any single model capability.
    - Impact: Careful engineers risk burnout and marginalization as high-volume agent users appear more productive despite shallow system understanding.
    - Watch next: Organizations add guardrails: agent limits, review budgets, required ownership of key code, and style guides to curb overload.
