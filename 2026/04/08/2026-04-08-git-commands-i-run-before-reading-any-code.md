# Git commands I run before reading any code

- Score: 1738 | [HN](https://news.ycombinator.com/item?id=47687273) | Link: https://piechowski.io/post/git-commands-before-reading-code/

## TL;DR

The author uses five git one-liners to “x-ray” a repo before opening any files:  
1) list most-changed files over the last year to spot churn hotspots;  
2) `git shortlog` to infer bus factor and contributor turnover;  
3) grep commit messages for “fix/bug/broken” to find bug-prone files;  
4) commits-per-month to see whether velocity is steady, surging, or fading;  
5) grep recent history for “revert/hotfix/rollback” to gauge firefighting frequency.  
HN likes the idea but warns these metrics are noisy without context, good commit hygiene, and filters for autogenerated or trivial files.

---

## Comment pulse

- Metrics are suggestive, not definitive → churn and shortlog can mislead when commits are tiny, noisy, automated, or the project is simply stable.  
- Commit-message quality is pivotal → squash-merge plus well-reviewed PR descriptions work better than relying on individual commits—counterpoint: many orgs never enforce this.  
- Tooling ergonomics matter → people alias long git/jj pipelines; some dislike “programmable VCS” complexity and prefer plain git + Unix pipes.

## LLM perspective

- View: Treat these commands as triage heuristics, not ground truth; always sanity-check against team history and code inspection.  
- Impact: Most useful for consultants or newcomers trying to prioritize attention in large or legacy codebases.  
- Watch next: Better tooling that combines churn, complexity, and ownership data into visual dashboards, with filters for generated code and bots.
